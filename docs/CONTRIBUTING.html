Contributing

Working conventions for the Renata codebase. See README.md for the project overview and docs/ARCHITECTURE.md for the design.

Development environment

make setup            # .venv + requirements
make demo             # verify the full data chain runs offline
make test             # run the suite

Python 3.9+ is the target. Note that some modules use PEP 604 union annotations (str | None); on 3.9 these require from __future__ import annotations at the top of the file. Follow the existing pattern.

Project layout and the rules that matter

  • Scoring is transparent. Every point in pipeline/score.py maps to a reason code. Do not add a scoring factor without a plain-English reason string.
  • One source, one agent. Adding a data source means a new SourceAgent subclass in agents/sources.py plus one line in REGISTRY. Do not fold source logic into the pipeline.
  • Serving is read-only. Code under api/ may only SELECT from the spine tables. The only writes are telemetry to events (and, later, outreach_drafts). New endpoints take the entitlement() dependency for the free/paid gate.
  • Numbers come from data, never from the model. The drafting and (future) proposal generators inject pre-computed, source-tagged values; the LLM writes prose around them.
  • Column drift is fixed in one place. If a 5500 load warns about a missing column, update the mapping in config.py against the DOL File Layout for that year. Required columns fail loud by design; do not relax that.

Database

Local development uses SQLite (renata.db, gitignored). Production targets Supabase Postgres via DATABASE_URL (direct, for ingest) and DATABASE_URL_POOLED (pooled, for the API). The schema in schema.sql is written to apply on both engines.

Tests

make test                       # all API tests
.venv/bin/python -m pytest tests/api/test_masking.py -q   # a single module

Tests run against a temporary SQLite database seeded from schema.sql (see tests/api/conftest.py). Do not weaken a test to make a change pass; fix the change.

Branching and commits

  • Branch off main; do not commit directly to main.
  • Keep commits scoped and the message imperative. Run make test before opening a PR.
  • The serving layer and the ingest layer are independent; prefer changes that respect that boundary so they can be reviewed and deployed separately.

Secrets

Never commit secrets. Configuration goes in .env (gitignored); .env.example documents the keys. Card data and PHI never touch application logs or the database (see the compliance notes in docs/).
