Architecture & Infrastructure

Strategic resolve: dedicated agents own the data sources

Every recommended data source gets its own dedicated agent with isolated access, cadence, and health. This is the resolution to "make sure we actually have agents on the sources we said we'd pull from." The registry:

Agent Source Access Cadence Freshness SLA Trigger it emits
form5500 DOL EFAST2 "Latest" datasets bulk download monthly 45 days foundation: renewal, incumbent, spend
rate_filing State DOI / SERFF (open states first) portal/API weekly 10 days carrier raised rates (strongest)
formd SEC EDGAR Form D public API daily 2 days just raised — new capacity
warn State WARN portals scrape daily 2 days workforce in flux
jobs postings provider/scrape API weekly 14 days outgrowing current plan

Each agent inherits one contract (agents/base_agent.py): fetch() → normalize() → common Signal schema, with shared retry/backoff, an idempotent write, and a heartbeat to source_health. Isolation is the point — one source erroring backs off and flags itself; the others keep running and the score degrades gracefully instead of silently.

The three infrastructure EBI passes

Pass 1 — Isolation & resilience

Even better if one bad source can't poison the system. - Split monolithic ingest into dedicated per-source agents (done — agents/). - Exponential backoff + bounded retries per agent; failures are caught and isolated. - source_health table records last run, ok/fail, signal count, and error per source. - Idempotent upserts (existing) so re-runs are safe; 5500 uses the monthly delta feed.

Pass 2 — Observability & honest freshness

Even better if the UI's "live / last refreshed / lead quality" reflects reality. - Every agent emits a heartbeat + freshness SLA; freshness_hours() exposes staleness. - system_temperature() derives lead volume, hot-lead ratio, and source health into the quality readout — so the landing page's temperature and "last refreshed" are computed from real state, never hardcoded. (When a feed goes stale, the dashboard should say so.) - Run ledger: agent outcomes logged to events for auditing and the Monday dossier.

Pass 3 — Scale, cost & extensibility

Even better if growth and new sources are cheap. - Per-source cadence scheduling instead of one big cron — 5500 monthly, WARN/Form D daily, rate filings/jobs weekly. In production, a queue/scheduler (e.g., cron+worker or a task queue) runs each agent on its own clock. - Separate batch ingest from realtime serving — heavy 5500 loads run offline; the score and Radar serve from precomputed tables, so the app stays fast and cheap. - Config-driven source registry — adding a source is a new SourceAgent subclass + one line in REGISTRY, not a rewrite. - Per-agent secret isolation — each agent reads only its own scoped credentials, so a leak or revocation is contained to one source.

Serving (next)

The frontend reads three things, all precomputed: the ranked Radar per metro, the reason codes per account, and the temperature/freshness snapshot. Auth + Stripe gate the unlock; the free preview is the same query with names masked. Keeping serving read-only over precomputed tables is what makes the "live dashboard" both honest and inexpensive.
