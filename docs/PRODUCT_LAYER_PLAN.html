Renata — Product Layer Plan

Sequencing the product layer onto the finished data spine. Produced via a four-expert strategic-resolve session (CTO, PM, COO, Legal) on 2026-06-13. This is the build order of record; the operating model and gates live in BUILD_PLAN.md, the infra design in ARCHITECTURE.md.

The core insight

The SQLite -> Postgres port is not part of "live DOL ingest." It is the substrate every other track sits on, so it is pulled out as Phase 1 (Track 0). Once it lands, the read path (serving API) and the write path (ingest) are independent and build in parallel, because serving is read-only over precomputed tables. That rule also means live data hot-swaps under a stable API — so live DOL ingest never blocks launch.

Decisions locked (owner sign-off)

  • Pricing (Gate 3): one Stripe product, two prices shown side by side — a higher month-to-month rate and $399/mo when billed annually (the transparent "good deal"). No per-win billing at launch; no win-detection infra required.
  • CRM integration: ships as a product feature (sync accounts/scores/drafts into the broker's CRM, read back stage changes), sequenced post-launch (Phase 6). The sync is what later enables an optional, opt-in success/performance add-on. CRM/win-detection is never a gate to paying.
  • Live DOL ingest: runs as a parallel feature, never a launch gate.
  • The in-app easy button is the headline wow: leads surface, a drafted message and a full multi-touch follow-up sequence wait for approve/edit. Elevated ahead of billing in the sequence so it can be demoed and hand-sold while Stripe is wired around it.
  • Build context: solo founder, Cursor, Austin/TX beachhead.

Phase 1 hardening — done 2026-06-13

All verified on SQLite (the local-dev fallback); the live Supabase port awaits DATABASE_URL.

  • Postgres-ready engine (ingest/load_5500.py get_engine): detects Postgres, sets pool_pre_ping=True; role="serving" prefers DATABASE_URL_POOLED (PgBouncer pooled port) and role="ingest" uses DATABASE_URL (direct port), each falling back to the other.
  • Column-drift fail-loud (config.py MAIN_REQUIRED/SCH_A_REQUIRED + _require): a missing required column raises ColumnDriftError, aborts the load, and records source_health.ok=0 (verified) instead of silently loading null premiums.
  • Idempotent signal writes (agents/base_agent.py _write + uq_signals_natural unique index): ON CONFLICT DO NOTHING on (company_id, type, source, observed_at). Verified: 3 identical writes -> 1 row.
  • Postgres-safe freshness_hours: handles datetime-vs-string and naive timestamps.
  • Serving indexes: idx_scores_score, idx_companies_state_band.

Remaining Phase 1 (needs owner's Supabase DATABASE_URL)

  • Run schema.sql on Supabase; migrate scores.reason_codes TEXT -> JSONB and timestamps -> TIMESTAMPTZ during the port.
  • Grant the API role SELECT-only on companies/plans/signals/scores (write only to events/outreach_drafts) so the read/write wall is enforced by the database.
