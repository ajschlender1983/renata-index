Renata — Phase 2 Serving API (build blueprint)

Read-only FastAPI over precomputed tables. Implementation-ready. Produced 2026-06-13.

Ground rules (from the codebase)

  • Python 3.9.6 -> every API file starts with from __future__ import annotations.
  • Raw sqlalchemy.text(), same style as radar.py/score.py. No ORM.
  • Engine: from ingest.load_5500 import get_engine called as get_engine(role="serving") (prefers DATABASE_URL_POOLED, pool_pre_ping=True). Build ONE engine at app startup on app.state.engine, not per request.
  • Read-only over companies/plans/signals/scores/source_health; writes only to events (telemetry) and later outreach_drafts. DB role grant enforces the wall too.
  • reason_codes is a JSON string on SQLite, JSONB on Postgres after the port. One helper (parse_reason_codes) accepts both. Same for updated_at/last_run (string vs datetime).

Endpoints

GET /v1/radar — ranked, masked Renewal Radar

Params: state (default TX), metro, min_participants/max_participants (ge=0), band (enum <50|50-99|100-499|500+), renewal_within (1-12 months), limit (default 25, max 200), cursor (opaque keyset). Effective limit clamped server-side: min(limit, free_row_cap) when locked (free cap default 10). Keyset on (in_play_score DESC, company_id ASC); fetch eff+1 to know if next_cursor exists.

CRITICAL: the scores<->plans JOIN is one-to-many (a sponsor can have multiple plan rows). Use a ROW_NUMBER() OVER (PARTITION BY company_id ORDER BY total_premium DESC NULLS LAST, id) CTE to pick one plan per company, mirroring load_schedule_a's "largest-premium headline" rule. Without it, the row cap and cursor corrupt (a company straddles a page boundary).

Masking (project_radar_row): real in_play_score, reason_codes, est_annual_spend, carrier, renewal_month for everyone; name -> "Account locked" and incumbent_broker -> null when locked. Mask happens AFTER SQL; the masked name never enters the payload, so view-source reveals nothing. Free row cap and mask are server-side only.

GET /v1/accounts/{company_id} — full brief, 402 for free tier

If not unlocked -> raise HTTPException(402) BEFORE any DB read (no data leaves the server for the locked tier). Returns the full account + contributing_signals from the signals table. 404 if no scored account for the id.

GET /v1/temperature — public, cached 60s

No entitlement() dependency (it is the honest "live" proof for the landing page). Passthrough of system_temperature(engine) plus per-source freshness via SourceAgent.freshness_hours(). 60s in-process TTL cache (module-level, thread-locked); send Cache-Control: public, max-age=60.

The entitlement seam (Phase 3 plugs in here, nothing else changes)

One entitlement(request) -> Tier dependency. Phase 2: returns Tier(unlocked=False) for everyone (anon), so the whole API is functional and demoable before auth/Stripe exist. Every endpoint takes tier: Tier = Depends(entitlement). Phase 3 wires only _verify_supabase_jwt (PyJWT + SUPABASE_JWT_SECRET) and one users.plan_tier lookup; unlocked = plan_tier in ("monthly","annual").

Test assertions (the load-bearing ones)

  • Masked radar: every row name == "Account locked", incumbent_broker null, and the real seeded names appear NOWHERE in the raw JSON body.
  • Real values still present when masked (score, spend, reason_codes, carrier).
  • Free row cap enforced even with limit=200.
  • One row per company (the two-plan fixture appears once; higher-premium plan wins).
  • Keyset: no dup company_id across pages; garbage cursor -> 400.
  • 402 on brief for anon, with no company data in the body; 200 + full brief when unlocked.
  • Temperature public, shape correct, two calls within 60s share cached_at.
  • parse_reason_codes handles both a JSON string and a Python list.

Status — BUILT 2026-06-13

The api/ package and all three endpoints are implemented and verified: 13 tests pass (tests/api/), and the server runs live against SQLite (masked Radar, public temperature, 402-gated brief all confirmed via curl). Phase 3 wires auth into the single entitlement() seam; the Supabase port (Phase 1 remainder) swaps the engine with no API changes.

Follow-up before launch

  • Masked company_id is the sponsor EIN. A determined free user could reverse-look-up the EIN to the company, partially defeating the name mask. Before public launch, emit an opaque per-account token as the public id in masked responses (keep the EIN server-side, map token->EIN on unlock). Low effort, do it when auth lands.

Top gotchas

  1. The one-to-many scores/plans JOIN (use the ROW_NUMBER CTE).
  2. reason_codes type changes across the SQLite->Postgres port (handle str OR list).
  3. Build one engine at startup with role="serving"; keep every query parameterized via text() bindparams; never f-string user input into SQL.
