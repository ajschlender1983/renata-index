Renata

Renewal intelligence for employee-benefits brokers. Renata reads public benefits filings, fuses them into a per-company In-Play Score (which employer groups are about to switch brokers), and drafts the outreach. The product persona — the content voice, the chatbot, the in-app assistant — is also named Renata. Tagline: the oracle, and the easy button.

Status: Phases 0–3 implemented and verified. Python 3.9+. Test suite green (make test). Confidential and proprietary. Internal handoff document; not for distribution.

What it is

The signal that an employer is about to switch benefits brokers already sits in public filings, fragmented and ignored. Renata is the agent that fuses it into one transparent score and writes the opening, so the broker's only job is the one thing that closes: the right human touch at the right time.

The system is built around a one-directional data flow and a hard rule that the score explains itself — every point maps to a plain-English reason code.

dedicated source agents ──▶ signals + source_health ──▶ scoring (In-Play Score)
        │                                                      │
   DOL 5500 ingest ──▶ companies + plans ─────────────────────┘
                                                               ▼
                                          Renewal Radar generator + serving API
                                                               ▼
                                          frontend (landing, dashboard, easy button)

Each external data source has its own dedicated agent (agents/) that owns access, cadence, retry, normalization, and health. One source failing degrades the score gracefully and never takes down the others. System freshness and lead temperature are computed from real state (system_temperature()), never faked.

Quickstart

Prerequisites

  • Python 3.9 or newer
  • No external services required for local development (SQLite + a bundled synthetic fixture)

Setup

make setup            # creates .venv and installs requirements
# or, manually:
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

Run the data spine (10-second smoke test, no download)

A synthetic 5500 fixture ships in data/2023/ so the whole chain runs offline.

make demo             # schema → ingest → score → agents → radar

This builds renata.db, loads the TX fixture, computes In-Play Scores, runs the source agents and the temperature readout, and writes radar.html. Expect 2 TX health plans with 100% renewal-month and cost-per-head coverage, and a scored account with reason codes.

Individual steps are also available: make schema, make spine, make score, make agents, make radar.

Run the serving API

make api              # uvicorn api.main:app on :8000

The API is read-only over the precomputed tables. Endpoints: - GET /v1/radar — ranked, masked Renewal Radar (free preview hides company name + broker) - GET /v1/accounts/{id} — full account brief (402 for the locked tier) - GET /v1/temperature — public freshness/quality snapshot - POST /v1/accounts/{id}/draft, GET /v1/drafts, PATCH /v1/drafts/{id} — the easy button

For a local demo of the paid features without auth, set RENATA_DEV_UNLOCK=1:

RENATA_DEV_UNLOCK=1 make api

Run the tests

make test             # pytest tests/api

Configuration

Copy .env.example to .env. No secrets are committed.

Variable Purpose
DATABASE_URL Postgres (Supabase) direct connection for the batch loader. Blank ⇒ local SQLite.
DATABASE_URL_POOLED Postgres pooled (PgBouncer) connection for the serving API. Falls back to DATABASE_URL.
RENATA_DATA_DIR Where downloaded DOL datasets are extracted.
ANTHROPIC_API_KEY Optional. Enables the Claude drafting path; the template path runs without it.
RENATA_DRAFT_MODEL Drafting model id (default claude-opus-4-8).
SUPABASE_JWT_SECRET Auth seam (planned).
STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET Billing (planned).

Architecture and conventions

  • Scoring is transparent on purpose. Every point maps to a plain-English reason code (pipeline/score.py). Renata has to explain herself; keep it that way.
  • Each data source is a dedicated SourceAgent (agents/sources.py) with its own access, cadence, retry, and health. Adding a source is a new subclass plus one line in REGISTRY, never source logic folded into the pipeline.
  • Serving stays read-only over precomputed tables. The API only SELECTs; the only writes are telemetry to events. This is what lets live ingest hot-swap under a stable API later.
  • Column drift lives in one place (config.py). The 5500 column names are the one thing that shifts between filing years; the loader fails loud on a missing required column rather than silently loading nulls.
  • The masking mechanic (api/masking.py) hides identity (company name, incumbent broker) and always returns the real value (score, spend, reason codes). It is one server-side code path; the free row cap and the mask are never enforced on the client.

See docs/ARCHITECTURE.md for the infrastructure design and CLAUDE.md for working conventions.

Compliance guardrails (regulated domain — do not cross)

  • DOL 5500, WARN, and SEC Form D are public; SERFF rate-filing access varies by state (start with the open states, do not overpromise nationwide).
  • Outreach is drafted and human-sent, never bulk-automated. There is no system send path.
  • Renata runs transparently as an AI analyst paired with a real human face, never a fabricated person. See docs/GTM_LINKEDIN_PLAYBOOK.md and docs/PROPOSAL_GENERATION.md for the full posture and the items flagged for outside counsel.
